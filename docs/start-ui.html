Nana Assistant Workspace

Guide for the loopback REST API and browser workspace launched by nana start.

Quick start

nana start starts the repo automation loop and, unless --no-ui is supplied, also starts a local REST API and assistant workspace bound to 127.0.0.1. The process prints the resolved URLs when the UI is ready: 

[start-ui] API: http://127.0.0.1:17653
[start-ui] Web: http://127.0.0.1:17654

Open the printed Web URL in your browser. Keep the terminal process running while you operate the console. Start UI now reuses local warm-state artifacts under ~/.nana/start/ui/ so the first open after a restart can avoid a cold overview rebuild when nothing relevant changed.

nana repo onboard owner/repo
nana repo config owner/repo --repo-mode fork --issue-pick label --pr-forward approve
nana start

What you can do in the workspace

Attention

Work one operator inbox for approvals, findings, blocked runs, draft work items, and tracked-issue decisions. Sync GitHub from this surface when you need fresh review and reply work items.

Usage

Inspect token spend across NANA-managed sessions, roots, phases, and models without leaving the assistant workspace.

Repo tabs

Use repo-scoped Overview, Scouts, Config, and Controls tabs without splitting operational editing across multiple top-level pages.

Controls

Schedule repo-scoped tasks, manage scheduled launches, edit findings, review markdown imports, and keep tracked issue scheduling in one operational tab.

End-to-end operator flow

  1. Onboard and configure each target repo with nana repo onboard, nana repo config, or nana repo explain.
  2. Run bare nana start for the normal long-running loop, or nana start --cycles <n> for a bounded run.
  3. Open the printed Web URL and use Attention to triage blocked work, approvals, findings, draft work items, and tracked-issue decisions.
  4. Use Usage when you need token-spend visibility without leaving the browser workspace.
  5. Use the repo-level Scouts tab to inspect auto-recovered scout proposals; stale startup cleanups are retried once there with explicit cooldown metadata before any failed approval is shown.
  6. Open a repo page when you need repo-local Overview, Scouts, Config, or Controls.
  7. Use repo Controls for task scheduling, scheduled-task management, findings editing, and finding-import review.
  8. Use run detail, retry timing, logs, and the Sync GitHub control in Attention when verification, publication, or review feedback needs fresh state.
  9. Stop the loop from the terminal with Ctrl+C when you are done operating automation.

Loopback API reference

The Web URL calls the API URL printed by the same nana start process. Endpoints return JSON and are intended for local automation and workspace UX; keep them on loopback and do not expose the ports outside your machine.

EndpointUse
GET /api/v1/overviewGlobal totals, repo summaries, recent runs, work items, and HUD payload.
GET /api/v1/usageUsage breakdowns, top sessions, and a usage-data version for the current filter set. The Usage view refreshes on entry, on manual/filter actions, and periodically while that view stays visible. Live usage reads are backed by SQLite; older usage JSON files and thread-usage*.json artifacts are treated only as compatibility import inputs.
GET /api/v1/attentionRanked attention queue plus the single best next item and recommended command.
GET /api/v1/issuesCompatibility-only legacy tracked-issue queue feed retained for one migration cycle. The current SPA redirects old queue hashes into Attention filters instead of rendering a dedicated issues page.
GET /api/v1/investigationsCompatibility-only legacy investigations feed retained for one migration cycle. Repo task scheduling now lives under repo Controls, while already-created investigation work is triaged through Attention.
POST /api/v1/investigationsStart a new investigation for the current workspace or a managed repo checkout.
GET /api/v1/reviewsCompatibility-only legacy review-draft feed retained for one migration cycle. The current workspace surfaces these items as work-item attention tiles instead of a dedicated feedback page.
GET /api/v1/repliesCompatibility-only legacy reply-draft feed retained for one migration cycle. The current workspace surfaces these items as work-item attention tiles instead of a dedicated feedback page.
POST /api/v1/feedback/syncRefresh GitHub notifications so new review/reply work items appear in Attention. The current SPA calls this from the Sync GitHub button in the Attention summary area.
GET /api/v1/approvalsCompatibility-only legacy approvals feed retained for one migration cycle. The current SPA redirects old approvals hashes into Attention approval filters instead of rendering a dedicated approvals page.
GET /api/v1/reposList start-aware repo summaries.
GET /api/v1/repos/<owner>/<repo>/start-stateInspect one repo's start state and automation settings.
PATCH /api/v1/repos/<owner>/<repo>/settingsUpdate repo-mode, issue-pick, PR-forwarding, publish settings, scout policies, and scout schedules.
PATCH /api/v1/repos/<owner>/<repo>/issues/<number>Schedule, defer, reprioritize, or clear scheduling for a mirrored issue.
POST /api/v1/repos/<owner>/<repo>/issues/<number>/investigateStart an investigation from a tracked issue.
POST /api/v1/repos/<owner>/<repo>/issues/<number>/launch-workLaunch work for a tracked issue through the planned-work path, using the issue's resolved work type.
GET /api/v1/repos/<owner>/<repo>/scout-itemsList local scout proposals, recovery metadata, cooldown state, and their available actions.
POST /api/v1/repos/<owner>/<repo>/planned-itemsCreate planned work for a repo. Manual planned items require work_type.
GET /api/v1/work/runsList recent work runs.
GET /api/v1/work/runs/<run-id>Inspect run manifest/status detail for local and GitHub-backed work, including paused retry timing.
POST /api/v1/work/runs/<run-id>/syncRun the safe GitHub feedback-sync flow for a GitHub-backed run.
GET /api/v1/work/runs/<run-id>/logsList log files for a run.
GET /api/v1/work-itemsList queued, paused, draft, hidden, or actionable work items.
GET /api/v1/eventsServer-sent events stream that publishes only generated_at, overview_version, and the last known usage_version; the browser fetches full overview payloads on overview changes and refreshes Usage separately while the Usage view is active.

Ports and safety controls

  • Default ports are API 17653 and Web 17654; if a requested port is busy, Nana tries nearby ports and prints the resolved URLs.
  • Use --ui-api-port <port> or --ui-web-port <port> when you need predictable local ports.
  • Use --no-ui for headless automation runs or environments where opening a local browser workspace is not desired.
  • The latest UI runtime metadata is written under ~/.nana/start/ui/runtime.json for local inspection.
